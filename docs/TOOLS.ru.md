## Файловые операции

### read_file

Читает и возвращает содержимое указанного файла. Поддерживает текстовые файлы, изображения (PNG, JPG, GIF, WEBP, SVG, BMP) и PDF-файлы.

**Параметры:**

| Имя             | Тип    | Обязательный | Описание                                             |
| --------------- | ------ | ------------ | ---------------------------------------------------- |
| `absolute_path` | string | Да           | Абсолютный путь к файлу для чтения                   |
| `offset`        | number | Нет          | Номер строки (с 0) для начала чтения (для пагинации) |
| `limit`         | number | Нет          | Максимальное количество строк для чтения             |

**Пример:**

```json
{
  "absolute_path": "/home/user/project/src/index.ts"
}
```

**Для больших файлов:**

```json
{
  "absolute_path": "/home/user/project/large-file.log",
  "offset": 100,
  "limit": 50
}
```

---

### read_many_files

Читает несколько файлов за одну операцию. Эффективнее нескольких вызовов `read_file` для пакетных операций.

**Параметры:**

| Имя      | Тип      | Обязательный | Описание                              |
| -------- | -------- | ------------ | ------------------------------------- |
| `paths`  | string[] | Да           | Массив абсолютных путей к файлам      |
| `offset` | number   | Нет          | Смещение строк                        |
| `limit`  | number   | Нет          | Максимальное количество строк на файл |

---

### write_file

Записывает содержимое в файл, создавая его при отсутствии или перезаписывая при наличии. Автоматически создаёт родительские директории.

**Параметры:**

| Имя         | Тип    | Обязательный | Описание                           |
| ----------- | ------ | ------------ | ---------------------------------- |
| `file_path` | string | Да           | Абсолютный путь к файлу для записи |
| `content`   | string | Да           | Содержимое для записи в файл       |

**Пример:**

```json
{
  "file_path": "/home/user/project/src/utils.ts",
  "content": "export function greet(name: string): string {\n  return `Привет, ${name}!`;\n}"
}
```

---

### edit

Выполняет поиск и замену в существующих файлах. Требует точного совпадения строк, включая пробелы и отступы.

**Параметры:**

| Имя           | Тип     | Обязательный | Описание                                             |
| ------------- | ------- | ------------ | ---------------------------------------------------- |
| `file_path`   | string  | Да           | Абсолютный путь к файлу для редактирования           |
| `old_string`  | string  | Да           | Точный текст для замены (должен совпадать полностью) |
| `new_string`  | string  | Да           | Текст для замены                                     |
| `replace_all` | boolean | Нет          | Заменить все вхождения (по умолчанию: false)         |

**Важные замечания:**

- Включите минимум 3 строки контекста до и после целевого текста
- Точно совпадайте с пробелами и отступами
- Используйте `replace_all: true` для замены всех вхождений

**Пример:**

```json
{
  "file_path": "/home/user/project/src/app.ts",
  "old_string": "function oldName() {\n  return 'old';\n}",
  "new_string": "function newName() {\n  return 'new';\n}"
}
```

## Поиск и навигация

### glob

Быстрый поиск файлов по шаблону. Работает с кодовой базой любого размера. Возвращает пути к файлам, отсортированные по времени изменения.

**Параметры:**

| Имя       | Тип    | Обязательный | Описание                                              |
| --------- | ------ | ------------ | ----------------------------------------------------- |
| `pattern` | string | Да           | Glob-шаблон (напр., `**/*.ts`, `src/**/*.js`)         |
| `path`    | string | Нет          | Директория для поиска (по умолчанию - корень проекта) |

**Примеры:**

```json
// Найти все TypeScript файлы
{ "pattern": "**/*.ts" }

// Найти JavaScript файлы в директории src
{ "pattern": "src/**/*.js" }

// Найти все JSON файлы
{ "pattern": "*.json", "path": "/home/user/project/config" }
```

---

### grep_search

Поиск паттернов в содержимом файлов с использованием регулярных выражений. По умолчанию регистронезависимый.

**Параметры:**

| Имя       | Тип    | Обязательный | Описание                                  |
| --------- | ------ | ------------ | ----------------------------------------- |
| `pattern` | string | Да           | Регулярное выражение для поиска           |
| `path`    | string | Нет          | Директория или файл для поиска            |
| `glob`    | string | Нет          | Glob-шаблон для фильтрации файлов         |
| `limit`   | number | Нет          | Максимальное количество совпадающих строк |

**Примеры:**

```json
// Найти все объявления функций
{ "pattern": "function\\s+\\w+" }

// Найти TODO в TypeScript файлах
{ "pattern": "TODO|FIXME", "glob": "*.ts" }

// Найти импорты в конкретной директории
{ "pattern": "import.*from", "path": "src/components" }
```

---

### list_directory

Выводит список файлов и поддиректорий в указанной директории.

**Параметры:**

| Имя                      | Тип      | Обязательный | Описание                              |
| ------------------------ | -------- | ------------ | ------------------------------------- |
| `path`                   | string   | Да           | Абсолютный путь к директории          |
| `ignore`                 | string[] | Нет          | Glob-шаблоны для игнорирования        |
| `file_filtering_options` | object   | Нет          | Опции для gitignore/ollama-codeignore |

**Пример:**

```json
{
  "path": "/home/user/project/src",
  "ignore": ["node_modules", "*.test.ts"]
}
```

## Инструменты разработки

### python_dev

Комплексный инструмент для разработки на Python - управление проектами, виртуальными окружениями и выполнение Python-команд.

**Параметры:**

| Имя                 | Тип      | Обязательный | Описание                               |
| ------------------- | -------- | ------------ | -------------------------------------- |
| `action`            | string   | Да           | Действие для выполнения (см. ниже)     |
| `script`            | string   | Нет          | Путь к Python-скрипту для run          |
| `args`              | string[] | Нет          | Дополнительные аргументы               |
| `packages`          | string[] | Нет          | Пакеты для операций pip                |
| `directory`         | string   | Нет          | Рабочая директория                     |
| `venv`              | string   | Нет          | Путь к виртуальному окружению          |
| `python_path`       | string   | Нет          | Путь к Python-интерпретатору           |
| `timeout`           | number   | Нет          | Таймаут в миллисекундах (макс. 600000) |
| `test_pattern`      | string   | Нет          | Шаблон тестов для pytest               |
| `lint_config`       | string   | Нет          | Путь к конфигу линтера                 |
| `requirements_file` | string   | Нет          | Путь к requirements.txt                |
| `command`           | string   | Нет          | Пользовательская команда               |

**Доступные действия:**

| Действие        | Описание                           |
| --------------- | ---------------------------------- |
| `run`           | Выполнить Python-скрипт            |
| `test`          | Запустить pytest тесты             |
| `lint`          | Запустить pylint анализ кода       |
| `format`        | Запустить black форматирование     |
| `venv_create`   | Создать виртуальное окружение      |
| `venv_activate` | Получить команду активации venv    |
| `pip_install`   | Установить пакеты через pip        |
| `pip_list`      | Список установленных пакетов       |
| `pip_freeze`    | Сгенерировать requirements.txt     |
| `mypy`          | Запустить mypy проверку типов      |
| `custom`        | Выполнить пользовательскую команду |

**Примеры:**

```json
// Запустить Python-скрипт
{
  "action": "run",
  "script": "main.py",
  "args": ["--verbose", "input.txt"]
}

// Запустить pytest с паттерном
{
  "action": "test",
  "test_pattern": "tests/unit/",
  "args": ["-k", "test_auth"]
}

// Создать виртуальное окружение
{
  "action": "venv_create",
  "venv": ".venv"
}

// Установить пакеты
{
  "action": "pip_install",
  "packages": ["requests", "numpy"],
  "venv": ".venv"
}
```

### nodejs_dev

Инструмент для разработки на Node.js/JavaScript с поддержкой npm, yarn, pnpm и bun. Автоопределение менеджера пакетов по lock-файлам.

**Параметры:**

| Имя               | Тип      | Обязательный | Описание                                        |
| ----------------- | -------- | ------------ | ----------------------------------------------- |
| `action`          | string   | Да           | Действие для выполнения                         |
| `package_manager` | string   | Нет          | Менеджер пакетов (`npm`, `yarn`, `pnpm`, `bun`) |
| `script`          | string   | Нет          | Путь к скрипту для run                          |
| `args`            | string[] | Нет          | Дополнительные аргументы                        |
| `packages`        | string[] | Нет          | Пакеты для add/remove                           |
| `directory`       | string   | Нет          | Рабочая директория                              |
| `dev`             | boolean  | Нет          | Добавить как dev-зависимость                    |
| `global`          | boolean  | Нет          | Установить глобально                            |
| `timeout`         | number   | Нет          | Таймаут в миллисекундах                         |
| `script_name`     | string   | Нет          | Имя скрипта из package.json                     |
| `command`         | string   | Нет          | Пользовательская команда                        |
| `background`      | boolean  | Нет          | Запустить в фоновом режиме                      |

**Доступные действия:**

| Действие     | Описание                           |
| ------------ | ---------------------------------- |
| `run`        | Выполнить Node.js скрипт           |
| `install`    | Установить зависимости             |
| `add`        | Добавить пакеты                    |
| `remove`     | Удалить пакеты                     |
| `update`     | Обновить пакеты                    |
| `run_script` | Запустить скрипт из package.json   |
| `test`       | Запустить тесты                    |
| `build`      | Запустить сборку                   |
| `dev`        | Запустить dev-сервер               |
| `lint`       | Запустить линтер                   |
| `exec`       | Выполнить npx/yarn dlx команду     |
| `info`       | Показать информацию о пакете       |
| `list`       | Список установленных пакетов       |
| `outdated`   | Проверить устаревшие пакеты        |
| `audit`      | Проверка безопасности              |
| `clean`      | Удалить node_modules и lock-файлы  |
| `init`       | Инициализировать новый проект      |
| `custom`     | Выполнить пользовательскую команду |

**Примеры:**

```json
// Установить зависимости
{ "action": "install" }

// Добавить пакеты
{
  "action": "add",
  "packages": ["express", "lodash"],
  "dev": false
}

// Запустить dev-сервер в фоне
{
  "action": "dev",
  "background": true,
  "package_manager": "bun"
}

// Запустить пользовательскую npx команду
{
  "action": "exec",
  "command": "create-next-app",
  "args": ["my-app", "--typescript"]
}
```

## Git операции

### git_workflow

Полная интеграция git workflow с поддержкой GitHub и GitLab (включая self-hosted). Предоставляет все базовые git операции плюс создание MR/PR.

**Параметры:**

| Имя         | Тип    | Обязательный | Описание                                           |
| ----------- | ------ | ------------ | -------------------------------------------------- |
| `operation` | string | Да           | Git операция для выполнения (см. ниже)             |
| `args`      | object | Нет          | Аргументы, специфичные для операции                |
| `directory` | string | Нет          | Рабочая директория (по умолчанию - корень проекта) |

**Доступные операции:**

#### Базовые операции

| Операция | Описание                     | Ключевые аргументы                            |
| -------- | ---------------------------- | --------------------------------------------- |
| `status` | Проверить статус репозитория | `short`, `branch`                             |

## SSH инструменты

### ssh_connect

Подключается к удалённой машине по SSH и выполняет команды. Поддерживает как прямое подключение, так и сохранённые профили.

**Параметры:**

| Имя             | Тип    | Обязательный | Описание                                                   |
| --------------- | ------ | ------------ | ---------------------------------------------------------- |
| `profile`       | string | Нет\*        | Имя сохранённого SSH профиля (альтернатива host/user)      |
| `host`          | string | Нет\*        | Имя хоста или IP адрес (обязательно без profile)           |
| `user`          | string | Нет\*        | Имя пользователя SSH (обязательно без profile)             |
| `command`       | string | Нет          | Команда для выполнения на удалённом сервере                |
| `port`          | number | Нет          | Номер порта SSH (по умолчанию: 22)                         |
| `identity_file` | string | Нет          | Путь к файлу приватного SSH ключа                          |
| `password`      | string | Нет          | Пароль для аутентификации (не рекомендуется)               |
| `timeout`       | number | Нет          | Таймаут в миллисекундах (макс. 600000, по умолчанию 60000) |
| `description`   | string | Нет          | Краткое описание операции                                  |

\*Либо `profile`, ЛИБО (`host` + `user`) должны быть указаны.

**Примеры:**

```json
// Использование сохранённого профиля
{
  "profile": "production",
  "command": "docker ps"
}

// Прямое подключение с SSH ключом
{
  "host": "192.168.1.100",
  "user": "admin",
  "port": 2222,
  "identity_file": "~/.ssh/deploy_key",
  "command": "systemctl status nginx"
}

// Интерактивная оболочка (без команды)
{
  "profile": "dev-server"
}
```

**Примечания безопасности:**

- SSH подключения всегда требуют подтверждения пользователя
- Используйте аутентификацию по ключу (`identity_file`) вместо пароля когда возможно
- Пароли маскируются в выводе инструмента

## Память и знания

Ollama Code предоставляет два инструмента для хранения информации с разными целями:

### Сравнение инструментов

| Характеристика | `save_memory` | `model_storage` |
|----------------|---------------|-----------------|
| **Назначение** | Пользовательские факты и предпочтения | Внутренние данные AI |
| **Инициатор** | Только по явному запросу пользователя | AI самостоятельно |
| **Формат** | Markdown (читаемый человеком) | JSON (структурированный) |
| **Подтверждение** | ✅ Всегда требуется | ❌ Автоматически |
| **Операции** | Только добавление | set/get/delete/list/merge/batch |
| **TTL** | ❌ Нет | ✅ Да (автоудаление) |
| **Метаданные** | ❌ Нет | ✅ createdAt, version, tags |

---

### save_memory

Инструмент для сохранения пользовательских фактов и предпочтений в долгосрочную память. Используйте его, когда пользователь явно просит запомнить что-то важное.

#### Когда использовать ✅

1. **Пользователь явно просит запомнить**
   - "Запомни, что моего кота зовут Барсик"
   - "Пожалуйста, запомни: я предпочитаю табы вместо пробелов"
   - "Не забудь, что я работаю удалённо по пятницам"

2. **Пользователь сообщает личное предпочтение**
   - "Мой любимый язык программирования — Python"
   - "Мне нравится тёмная тема во всех редакторах"
   - "Я предпочитаю краткие ответы"

3. **Пользователь делится важным контекстом**
   - "Этот проект использует строгий TypeScript"
   - "Моя команда использует conventional commits"
   - "Мы следуем стайлгайду Airbnb"

#### Когда НЕ использовать ❌

- Для временных данных сессии (используйте `model_storage` с namespace `session`)
- Для структурированных данных вроде roadmap, метрик (используйте `model_storage`)
- Для больших текстов или кода (используйте файлы)
- Для данных с TTL/истечением срока (используйте `model_storage`)
- Для внутреннего обучения AI без запроса пользователя (используйте `model_storage`)

#### Параметры

| Имя     | Тип    | Обязательный | Описание                                              |
| ------- | ------ | ------------ | ----------------------------------------------------- |
| `fact`  | string | Да           | Факт для запоминания (чёткое, самодостаточное утверждение) |
| `scope` | string | Нет          | `global` (все проекты) или `project` (текущий проект). Если не указан — пользователь выберет |

#### Расположение файлов

| Scope | Путь | Назначение |
|-------|------|------------|
| `global` | `~/.ollama-code/OLLAMA_MEMORY.md` | Общее для всех проектов |
| `project` | `./OLLAMA_MEMORY.md` | Только текущий проект |

#### Примеры

**Базовое использование:**
```json
{
  "fact": "Мой любимый язык программирования — TypeScript"
}
```

**С указанием scope:**
```json
{
  "fact": "В этом проекте всегда используй pnpm, никогда npm",
  "scope": "project"
}
```

**Как выглядит файл памяти:**
```markdown
# Project Context

Некоторый существующий контент...

## Ollama Added Memories

- Мой любимый язык программирования — TypeScript
- В этом проекте всегда используй pnpm, никогда npm
- Я предпочитаю краткие ответы без лишних объяснений
- URL API для разработки: http://localhost:3000
```

### model_storage

Универсальное key-value хранилище для AI-модели с полной поддержкой CRUD-операций, TTL и метаданными. Используется AI для внутренних нужд без участия пользователя.

#### Когда использовать ✅

1. **Roadmap проекта и планы**
   - Milestone'ы и планирование функционала
   - Разбивка задач со статусами
   - Планирование версий

2. **Управление базой знаний**
   - Изученные паттерны и соглашения
   - Сниппеты документации API
   - Шаблоны кода

3. **Данные сессии**
   - Временное рабочее состояние
   - Контекст текущей задачи
   - Отслеживание прогресса

4. **Обучение и улучшения**
   - Исправления использования инструментов
   - Маппинг алиасов
   - Паттерны ошибок и решения

5. **Метрики и статистика**
   - Данные производительности
   - Паттерны использования
   - Отслеживание времени

#### Когда НЕ использовать ❌

- Для сохранения фактов по запросу пользователя (используйте `save_memory`)
- Для данных, которые пользователь хочет редактировать вручную (Markdown дружелюбнее)

#### Пространства имён (Namespaces)

| Namespace | Назначение | Режим по умолчанию | Примеры ключей |
|-----------|------------|-------------------|----------------|
| `roadmap` | Планы проекта, milestone'ы | persistent | `v1.0`, `v2.0`, `q1-goals` |
| `session` | Временные данные сессии | session | `current-task`, `temp-state` |
| `knowledge` | Изученные факты, паттерны | persistent | `api-pattern`, `auth-flow` |
| `context` | Контекст текущей задачи | session | `active-feature`, `decisions` |
| `learning` | Исправления инструментов, алиасы | persistent | `tool-fix`, `command-alias` |
| `metrics` | Статистика, производительность | persistent | `daily-stats`, `response-times` |

#### Режимы хранения

| Режим | Хранилище | Жизненный цикл | Использовать для |
|-------|-----------|----------------|------------------|
| `persistent` | JSON файлы | Переживает перезапуски | roadmap, knowledge, learning |
| `session` | Только память | Очищается при выходе | временные данные, context |

**Автоопределение:** Namespace'ы `session` и `context` по умолчанию используют session mode. Остальные — persistent.

#### Параметры

| Имя              | Тип      | Обязательный | Описание                                               |
| ---------------- | -------- | ------------ | ------------------------------------------------------ |
| `operation`      | string   | Да           | Операция для выполнения                                |
| `namespace`      | string   | Да           | Пространство имён                                       |
| `key`            | string   | Нет\*        | Ключ для сохранения/получения                          |
| `value`          | any      | Нет\*        | Значение (любой JSON-сериализуемый тип)                |
| `persistent`     | boolean  | Нет          | Переопределить автоопределение режима                  |
| `scope`          | string   | Нет          | `global` или `project`                                 |
| `ttl`            | number   | Нет          | Время жизни в секундах (автоудаление)                  |
| `tags`           | string[] | Нет          | Теги для категоризации                                 |
| `includeMetadata`| boolean  | Нет          | Включить метаданные в результаты get/list              |
| `actions`        | array    | Нет          | Массив операций для batch                              |

\*Обязательно для операций `set`, `get`, `delete`, `append`, `merge`.

#### Операции

| Операция | Описание |
|----------|----------|
| `set` | Сохранить значение (перезаписывает существующее) |
| `get` | Получить значение по ключу |
| `delete` | Удалить ключ |
| `list` | Список всех ключей в namespace |
| `append` | Добавить элемент в массив |
| `merge` | Объединить объект с существующим |
| `clear` | Очистить все данные в namespace |
| `exists` | Проверить существование ключа |
| `stats` | Получить статистику хранилища |
| `batch` | Выполнить несколько операций атомарно |

**set — Сохранить значение:**
```json

  "namespace": "roadmap",
  "key": "v1.0",
  "value": {
    "features": ["auth", "dashboard"],
    "status": "in-progress",
    "targetDate": "2025-03-01"
  }
}
```

**set с TTL (истекает через 1 час):**
```json

  "operation": "set",

  "key": "temp-cache",
  "value": "временные данные",

```


**get — Получить значение:**

**get с метаданными:**

```json
{
  "operation": "append",
  "namespace": "knowledge",
  "key": "learned-patterns",
  "value": "Всегда проверять null перед доступом к вложенным свойствам"
}
```

**merge — Объединить объекты:**
```json
{
  "operation": "merge",
  "namespace": "knowledge",
  "key": "api-conventions",
  "value": {
    "errorFormat": "RFC7807",
    "authHeader": "Bearer token"
  }
}
```

**batch — Множественные операции:**
```json
{
  "operation": "batch",
  "namespace": "roadmap",
  "actions": [
    { "operation": "set", "key": "v1.0", "value": {"status": "done"} },
    { "operation": "set", "key": "v2.0", "value": {"status": "planning"} },
    { "operation": "delete", "key": "deprecated-feature" }
  ]
}
```

**stats — Статистика namespace:**
```json
{
  "operation": "stats",
  "namespace": "roadmap"
}
```

#### Scope (только для persistent storage)

| Scope | Расположение | Использование |
|-------|--------------|---------------|
| `global` | `~/.ollama-code/storage/` | Общее для всех проектов |
| `project` | `<project>/.ollama-code/storage/` | Только текущий проект |

```json
{
  "operation": "set",
  "namespace": "roadmap",
  "key": "project-goals",
  "value": { "q1": "MVP релиз" },
  "scope": "project"
}
```

#### Метаданные

Каждая запись автоматически отслеживает:

```json
{
  "value": { ... },
  "metadata": {
    "createdAt": "2025-01-15T10:30:00Z",
    "updatedAt": "2025-01-16T14:20:00Z",
    "version": 3,
    "ttl": 3600,
    "expiresAt": "2025-01-15T11:30:00Z",
    "tags": ["auth", "security"],
    "source": "session"
  }
}
```

#### Расположение хранилища

**Persistent:**
```
~/.ollama-code/storage/
├── roadmap.json
├── knowledge.json
├── learning.json
└── metrics.json

<project>/.ollama-code/storage/
├── roadmap.json
└── knowledge.json
```

**Session:** Хранится в памяти, очищается при завершении сессии.

### Лучшие практики

**Для save_memory:**
1. Держите факты краткими — одно чёткое утверждение
2. Будьте конкретны — "Я использую 2 пробела для YAML" вместо "У меня есть предпочтения отступов"
3. Выбирайте scope wisely — используйте `project` для специфичных настроек проекта
4. Периодически просматривайте — файл памяти можно редактировать вручную
5. Не дублируйте — проверьте, не сохранён ли уже факт

**Для model_storage:**
1. Используйте подходящие namespace — организуйте данные по назначению
2. Устанавливайте TTL для временных данных — предотвращает накопление устаревших данных
3. Используйте теги для фильтрации — упрощает поиск связанных записей
4. Batch операции эффективнее — для множественных изменений
5. Проверяйте перед set — используйте `exists` чтобы случайно не перезаписать

## Обработка ошибок

Все инструменты возвращают согласованную структуру ошибки при возникновении проблем:

```typescript
{
  "llmContent": "Подробное сообщение об ошибке для LLM",
  "returnDisplay": "Дружелюбное сообщение об ошибке для пользователя",
  "error": {
    "message": "Детали ошибки",
    "type": "код_типа_ошибки"
  }
}
```

**Типичные типы ошибок:**

| Тип ошибки            | Описание                             |
| --------------------- | ------------------------------------ |
| `file_not_found`      | Запрошенный файл не существует       |
| `permission_denied`   | Недостаточно прав                    |
| `invalid_tool_params` | Неверные или отсутствующие параметры |
| `execution_failed`    | Ошибка выполнения инструмента        |
| `timeout`             | Превышено время ожидания             |

## Лучшие практики

### Файловые операции

1. Всегда используйте абсолютные пути
2. Читайте файлы перед редактированием для понимания контекста
3. Включайте достаточный контекст в `old_string` для редактирования
4. Используйте `glob` или `grep_search` перед чтением для поиска файлов

### Поиск и навигация

1. Используйте `glob` для поиска файлов по шаблону имени
2. Используйте `grep_search` для поиска содержимого файлов
3. Комбинируйте инструменты для эффективного исследования
4. Используйте glob-фильтры для сужения области поиска

### Инструменты разработки

1. Используйте специализированные инструменты (`python_dev`, `nodejs_dev`, `golang_dev`) вместо сырых shell-команд
2. Устанавливайте `background: true` для длительных процессов
3. Настраивайте соответствующие таймауты для операций сборки

### Управление задачами

1. Создавайте списки задач для многошаговых задач
2. Обновляйте статус в реальном времени
3. Отмечайте задачи завершёнными сразу после выполнения
4. Имейте только одну задачу со статусом `in_progress` одновременно
